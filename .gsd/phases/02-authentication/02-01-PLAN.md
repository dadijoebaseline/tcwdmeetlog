# Phase 2, Plan 1: Authentication & Role Management

## Objective

Implement a complete, production-ready authentication system that supports Gmail sign-up/login, role-based access control (RBAC), user profile management, and protected routes.

## Context

Phase 1 created the foundation. Phase 2 builds on it by:
- Enhancing the basic Gmail auth from Phase 1
- Implementing proper role management with Firestore
- Creating user profiles with editable information
- Setting up protected routes and middleware
- Adding logout and session management
- Implementing role-based redirects

## Tasks

### Task 1: Enhance Authentication Context
**Type:** `auto`

Improve the auth context to handle role-based claims and permissions.

**Work:**
- Update `lib/auth-context.tsx` to fetch user role from Firestore custom claims
- Add role checking utilities
- Implement user profile fetching
- Add loading states for role resolution
- Handle role transitions (user changing role)

**Verification:**
- [ ] useAuth hook returns user, role, loading state
- [ ] Role is fetched from Firestore correctly
- [ ] Auth state persists across page refreshes
- [ ] No console errors

---

### Task 2: Create Protected Route Middleware
**Type:** `auto`

Implement route protection based on user authentication status and role.

**Work:**
- Create `lib/route-guard.tsx` - Client-side route protection component
- Create `lib/api-auth.ts` - API route protection utilities
- Handle redirect logic for unauthenticated users
- Implement role-based route access (HR-only, Attendee-only)
- Create hook: `useRequireAuth(role?)`

**Verification:**
- [ ] Unauthenticated users redirected to login
- [ ] HR-only routes block attendees

- [ ] Redirect includes next URL for post-login redirect
